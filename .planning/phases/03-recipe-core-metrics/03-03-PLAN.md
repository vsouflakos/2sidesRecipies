Method `merge(array ...$allergenSets): array` — each set is `array<array{slug: string, state: string}>`; returns a `slug => state` map where `contains` always wins over `may_contain` (per RESEARCH.md code example — copy that logic).     Create `app/Support/Recipes/CircularReferenceDetector.php` in `App\Support\Recipes`. Method `wouldCreateCycle(int $parentRecipeId, int $candidateRecipeId): bool` implementing a full BFS/DFS over the sub-recipe graph (RESEARCH.md Pattern 3 — Pitfall 2 warns a direct-self-check is NOT enough; the 3-node cycle A->B->C->A must be caught). Start from `$candidateRecipeId`; for each visited recipe, find the recipe IDs it references via sub-recipe lines — query `RecipeIngredientLine::whereNotNull('sub_recipe_version_id')` joined to `RecipeVersion` to resolve each `sub_recipe_version_id` back to its `recipe_id`; enqueue those. If `$parentRecipeId` is ever reached, or `$parentRecipeId === $candidateRecipeId`, return true. Track a `$visited` set to avoid infinite loops on pre-existing cycles.

    Create `app/Support/Recipes/RecipeDraftManager.php` in `App\Support\Recipes`. Methods:
    - `applyEdit(RecipeDraft $draft, string $action, array $newData): void` — inside a DB transaction: insert a `RecipeDraftEdit` row capturing `before_snapshot = $draft->data` and `sequence = $draft->edit_sequence`, then set `$draft->data = $newData`, increment `$draft->edit_sequence`, save. The `before_snapshot` enables Recall.
    - `recall(RecipeDraft $draft, int $expectedSequence): array` — Pitfall 5: if `$expectedSequence !== $draft->edit_sequence`, throw an `App\Exceptions\DraftSequenceMismatchException` (create this exception class; the controller maps it to a 409). Otherwise, inside a transaction: take the most recent `RecipeDraftEdit` (highest sequence), restore `$draft->data = $edit->before_snapshot`, decrement `edit_sequence`, delete that edit row, save. Return the restored `data`. If the edit log is empty, return `$draft->data` unchanged.

    Create `app/Support/Recipes/AggregatedMetrics.php` — readonly value object holding the prepared per-line array plus portion/yield context (`__construct(public array $lines, public BigDecimal $portions, public BigDecimal $totalYieldG)`).

    Create `app/Support/Recipes/MetricsAggregator.php` in `App\Support\Recipes`. Method `prepareLines(RecipeDraft $draft): AggregatedMetrics`. For each line in `$draft->data['sections'][*]['lines']`:
    - Ingredient line: resolve grams via `GramNormalizer`, attach the ingredient's nutrition values, `per_gram_cost` (latest user price), allergen set, yield_pct, is_flour_base.
    - Sub-recipe line: load the pinned `RecipeVersion`; compute `scaleFactor = BigDecimal::of(lineGramWeight)->dividedBy(BigDecimal::of($version->yield_g), 10, RoundingMode::HALF_UP)` — Pitfall 4: assert `yield_g` is non-null before dividing; throw a clear exception if null. Scale the version's `cached_nutrition_json` and `cached_cost_per_gram` by `scaleFactor`; carry `cached_allergen_slugs` through unchanged. Produce a synthetic "line" whose `grams`, nutrition, and `per_gram_cost` already reflect the scaled contribution so the downstream calculators treat a sub-recipe identically to an ingredient.
    The output `AggregatedMetrics.lines` array is the exact `$lines` shape the Plan 02 calculators consume.
