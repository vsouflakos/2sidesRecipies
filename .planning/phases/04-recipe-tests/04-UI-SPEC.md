Inherited from Phase 1 contract. All values multiples of 4. Applied via Tailwind utilities through the `@theme` block in `resources/css/app.css`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge internal padding, rating star gaps |

| md | 16px | Card padding, modal form field spacing, between-section gaps |
| lg | 24px | Modal section padding, tests page section breaks |

## Color

Inherited from Phase 1 contract. No new CSS variables needed. Existing warm-minimal token set applies directly.

### 60/30/10 Split (unchanged from Phase 1)

| Role | Token | OKLCH (light) | Usage in this phase |
|------|-------|---------------|---------------------|
| Dominant (60%) | `--background` | `oklch(0.981 0.007 60)` | Tests page background, modal surface, photo grid background |
| Secondary (30%) | `--secondary` / `--muted` | `oklch(0.948 0.010 60)` | Test card surfaces, change-row stripe, rating dimension row alternate background, empty state area |
| Accent (10%) | `--accent` | `oklch(0.550 0.035 60)` | See reserved-for list below |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` | Delete test button / confirmation only |

**Accent (`--accent`) is reserved for this phase:**
1. The currently selected type toggle tab (Trial / Experiment) — active tab gets `bg-accent text-accent-foreground`
2. The overall rating numeric value display when a test record is in focus in the test card
3. Verdict badge background for "Worked" verdict only (positive outcome signal)

Accent is NOT used for: primary buttons, test card hover state, photo thumbnails, any general hover state.

**Verdict badge color mapping:**

| Verdict | Badge variant | Rationale |
|---------|--------------|-----------|
| Worked | `bg-accent text-accent-foreground` | Warm positive — consistent with accent reservation |
| Didn't work | `variant="outline"` with `text-destructive border-destructive` | Clear negative signal |
| Inconclusive | `variant="secondary"` | Neutral — no strong signal |

*Source: Phase 1 UI-SPEC (inherited). Accent reservation extension and verdict badge mapping are Claude's discretion.*

## Surface Contracts

### Surface 1: Tests Index Page

**Route:** `GET /recipes/{recipe}/tests`
**File:** `resources/js/pages/recipes/tests/index.tsx`
**Layout:** Same app shell (sidebar + content area) as all other recipe pages.

**Page structure (top to bottom):**
1. Breadcrumb: "Recipes → {Recipe Name} → Tests" — using existing breadcrumb pattern
2. Page heading: "Tests for {Recipe Name}" (Display 28px/600) + "Back to recipe" ghost link aligned right
3. Stats bar: "{N} tests recorded" (Label/muted) + "Latest score: {N}/10" (Label/muted) — hidden when no tests
4. Primary CTA: "Record test" button (default variant, `PlusIcon` prefix) — opens `TestRecordModal`
5. Tests list: chronological descending (`tested_at DESC`), each test as a `TestCard`
6. Empty state (no tests yet): see Copywriting Contract

**Loading state:** Skeleton — 3 `TestCard`-shaped skeleton blocks, showing card outline + content placeholders.

**No filtering in Phase 4.** Tests page is chronological list only; filtering deferred.

---

### Surface 2: TestCard Component

**File:** `resources/js/components/recipes/test-card.tsx`

**Card anatomy (top to bottom within card):**

1. **Card header row:**
   - Left: Type badge (`Trial` in `variant="secondary"` or `Experiment` in `variant="outline"`) + Version label ("v{N}" in Label/muted)
   - Right: Tested date (Label/muted, e.g. "16 May 2026") + action dropdown (ellipsis icon, `DropdownMenu` with Edit / Delete)

2. **Overall score display:**
   - Large display: "{N}/10" — Heading size (20px/600) + accent foreground when score ≥ 7, muted-foreground when score < 7
   - Row of 4 default dimension scores if present (Label/14px, each "Taste: {N}" pattern)

3. **Verdict row (experiment type only):**
   - Verdict badge (see verdict badge color mapping in Color section)
   - Inline: "Hypothesis:" label (Label/muted) + first line of hypothesis text truncated to 1 line

4. **Tasting notes preview:**
   - `line-clamp-2` — 2 lines max, Body/16px/400
   - "Read more" link if truncated (ghost variant, opens full view)

5. **Photo strip (if photos exist):**
   - Horizontal row of up to 4 thumbnail `img` elements (80×80px, `object-fit: cover`, `rounded-md`)
   - "+{N} more" label if more than 4 photos (Label/muted)
   - Click any thumbnail → opens `TestPhotoGrid` lightbox

**Card hover state:** `bg-muted` (warm light grey), cursor-pointer on the tasting notes area.

### Surface 3: TestRecordModal

**File:** `resources/js/components/recipes/test-record-modal.tsx`

**Trigger:** "Record test" button on tests page, or "Edit" from test card dropdown.

**Modal width:** `max-w-2xl` (consistent with existing Phase 3 dialogs).

**Modal header:**
- Title: "Record test" (new) or "Edit test" (edit mode) — Display size via `DialogTitle`
- Description: "Log a trial run or structured experiment against a specific recipe version." — Label/muted via `DialogDescription`

**Form sections in order:**

**Section 1 — Type & Version (always visible)**
| Field | Control | Notes |
|-------|---------|-------|
| Test type | `ToggleGroup` (single): "Trial run" / "Experiment" | Default: "Trial run". Selecting Experiment reveals Section 4. |
| Recipe version | `Select` | Options: all versions newest-first, labeled "v{N} — {committed_at date}". Default: current version. |
| Tested on | `Input type="date"` | Default: today. Required. |

**Section 2 — Feedback (always visible)**
| Field | Control | Notes |
|-------|---------|-------|
| Overall rating | `Input type="number" min=1 max=10` | Required. Label: "Overall rating (1–10)". |
| Tasting notes | `Textarea` rows=3 | Optional. Placeholder: "Describe the taste, texture, and any observations…" |

**Section 3 — Ratings (always visible)**
- Heading: "Per-dimension ratings" (Heading/20px/600)
- 4 default dimensions pre-rendered as `RatingDimensionRow`: Taste, Texture, Appearance, Aroma
- Each row: dimension name (Label/14px) + `Input type="number" min=1 max=10` (optional) — score can be left blank
- "Add dimension" ghost button at bottom of list — appends a custom `RatingDimensionRow` with editable name input
- Custom dimension rows show a remove (×) button; default dimension rows do not

**Section 4 — Experiment (visible only when type = Experiment)**
- Heading: "Experiment details" (Heading/20px/600) — appears below ratings
| Field | Control | Notes |
|-------|---------|-------|
| Hypothesis | `Textarea` rows=2 | Required when type = experiment. Placeholder: "What were you testing or changing?" |
| Outcome narrative | `Textarea` rows=2 | Optional. Placeholder: "What actually happened?" |
| Verdict | `Select` | Options: "Worked", "Didn't work", "Inconclusive". Optional. |
| What changed | Structured change rows (see below) | Optional. |

**Change rows UI:**
- Each row: three `Input` fields side by side — "What changed" / "Expected effect" / "Actual effect" — all with `text-sm` placeholder
- "Add change" ghost button appends a new row
- Each row has a remove (×) button
- Max 20 rows (matches validation)

**Section 5 — Photos (always visible)**
- Heading: "Photos" (Heading/20px/600)
- Upload area: dashed border box (`border-dashed border-2 border-border`), height 80px, centered text: "Drag photos here or click to browse" (Label/muted), `CloudUploadIcon` prefix
- Below upload area: horizontal thumbnail grid of staged/existing photos (80×80px each, `rounded-md`, `object-fit: cover`)
- Each photo thumbnail: overlay × button appears on hover to remove from queue
- Photo count indicator: "{N}/8 photos" (Label/muted) — when limit reached, upload area shows "Photo limit reached" tooltip and disables interaction
- Accepted: JPEG, PNG, WebP. Max 5 MB each.
- Photos are NOT uploaded on selection — upload happens atomically on form submit.

**Modal footer:**
- Left: Character count for tasting notes ("0 / 5000") — Label/muted, visible when notes textarea is focused
- Right: "Cancel" (secondary button) + "Save test" (default button, shows `Spinner` while submitting)

**Validation inline errors:** Appear below the relevant field in Label/14px/`text-destructive`. Never use toast for field-level errors.

**Modal close behaviour:** Closing via Cancel or the × button without saving leaves no orphaned files (photos not yet uploaded; local preview state is discarded).

**File:** `resources/js/components/recipes/test-photo-grid.tsx`

**On saved test card:** Up to 4 thumbnails shown inline (80×80px). Click any thumbnail → opens lightbox.

## Accessibility Contract

- All interactive elements meet WCAG 2.5.5 (minimum 44×44px touch target).
- Photo remove (×) buttons carry `aria-label="Remove photo {N}"` for screen-reader identification.
- Photo thumbnail lightbox arrows carry `aria-label="Previous photo"` / `aria-label="Next photo"`.
- Rating dimension score inputs carry `aria-label="{Dimension} rating"` (e.g. `aria-label="Taste rating"`).
- Modal uses Radix Dialog's built-in focus trap and `aria-modal="true"`.
- Type toggle uses `role="group"` with `aria-label="Test type"`.
- Delete confirmation dialog: destructive confirm button is NOT auto-focused on open — focus lands on the dismiss ("Keep test") button per WCAG destructive action pattern.
- Version picker `Select` carries `aria-label="Recipe version for this test"`.
- Test card action dropdown trigger carries `aria-label="Test actions"`.
- Photo upload drop zone carries `role="button"` and `tabIndex={0}` so keyboard users can activate it.
