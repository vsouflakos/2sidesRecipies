## Surface Inventory

This phase adds four distinct UI surfaces.

### Surface 1: My Recipes card — publish state augmentation

The existing `RecipeCard` component gains a "Published" badge and a publish/unpublish menu item. No new card component is created.

- "Published" badge: `<Badge>` using `variant="secondary"` with an accent-adjacent warm tone to distinguish it from the cuisine badge. Rendered in the card body below the recipe name, only when `is_published = true`.
- Card menu (DropdownMenu already present from Phase 3): add "Publish recipe" item (when not published) and "Unpublish recipe" item (when published) alongside existing "Edit" and "Delete" items.
- "Delete" menu item disabled state (visually muted, tooltip or inline message) when `is_published = true`. Do NOT hide the item — show it as disabled with a tooltip: "Unpublish this recipe before deleting it."

Layout constraint: the badge sits on a single line between recipe name and cuisine badge. It does not wrap.

### Surface 2: Recipe builder — publish controls (augmentation of show.tsx)

The existing `show.tsx` page gains a publish control in the page header area (alongside the existing Save and Recall actions).

- When unpublished: a "Publish recipe" button (outline variant) in the header action group.
- When published: a "Published" badge + "Unpublish recipe" button (ghost variant) + conditional "Update to current version" inline cue.
- "Update to current version" cue: rendered as a callout banner (`bg-muted rounded-md px-4 py-3`) directly below the header action row, only when `is_published = true` AND `published_version_id !== current_version_id`. Contains the cue text and an "Update to current" button (`variant="outline" size="sm"`).
- The publish control area must not reflow the existing header layout. Use `flex items-center gap-2` within the existing action group.

### Surface 3: Public library index (`library/index.tsx`)

A new page mirroring `recipes/index.tsx` in structure but scoped to published recipes across all users.

Layout:
- Page heading: "Library" at 28px semibold, `leading-tight`.
- No "New Recipe" CTA (this is a read-only browse surface).
- Search input: identical to `recipes/index.tsx` — `SearchIcon` inset left, `w-full pl-9`.
- Filter bar: `RecipeFilters` component with filters for tag, cuisine, allergen, difficulty, max_total_time. The ingredient filter is EXCLUDED from the public library (it is a private-recipe planning tool; not relevant to public browse). Keep `RecipeFilters` itself unchanged — just do not pass the ingredient filter value or include `ingredient` in the query params for this route.
- Card grid: `grid-cols-1 gap-6 md:grid-cols-2 lg:grid-cols-3`. Cards use a `LibraryRecipeCard` variant (or `RecipeCard` with a `showAuthor` prop override) that:
  - Omits `cost_per_portion` (private data — must never appear)
  - Adds author name: `<span className="text-[14px] text-muted-foreground">by {author_name}</span>` rendered below the cuisine badge
  - Links to `library.show` (slug-based) not `recipes.show`
- Skeleton loading: identical structure and count (6 skeletons) as `recipes/index.tsx`.
- Pagination: identical component usage to `recipes/index.tsx`.
- Guest layout: the public library page uses a minimal guest header (logo + "Library" nav link + optional "Sign in" link) when `auth.user` is null. If `AppSidebarLayout` can handle null `auth.user` without crashing (to be verified in Wave 0 against `NavUser`), use it; otherwise use a `GuestLayout` wrapper.

### Surface 4: Public recipe show page (`library/show.tsx`)

A new, dedicated, clean page. NOT a reuse of `show.tsx`. No builder UI, no editor, no version picker, no tests panel, no AI chat.

Layout sections (top to bottom):

1. **Hero image** — full-width, `aspect-video`, `object-cover`. If no hero image, render a placeholder with `UtensilsCrossedIcon` centered on `bg-muted`.
2. **Recipe header block** — `py-8 px-4 md:px-8`:
   - Recipe name: 28px semibold, `leading-tight`
   - Author line: "by {author_name}" at 14px, `text-muted-foreground`
   - Published date: 14px, `text-muted-foreground`, format "Published DD MMM YYYY"
   - Cuisine badge (`variant="secondary"`) + difficulty badge (`variant="outline"`) + time metadata on one flex row, `gap-2`
3. **Metrics strip** — `border-y py-4 px-4 md:px-8`, `flex flex-wrap gap-6`:
   - Calories per portion: "X kcal / portion" at 14px
   - Allergen icons row (EU 14-allergen, `AllergenIcons` component)
   - NO cost, NO selling price, NO food cost %
4. **Sections + ingredient lines** — rendered from `snapshot.sections`, padded `px-4 md:px-8 py-6`:
   - Section heading: 20px semibold
   - Each ingredient line: 16px, `flex gap-2`, quantity + unit + ingredient name. No pricing, no cost fields.
5. **Steps** — ordered list, `px-4 md:px-8 py-6`:
   - Step number: 20px semibold
   - Step text: 16px, `leading-relaxed`
6. **Nutrition panel** — `Tabs` with two tabs: "Per portion" and "Per 100 g". Renders from `cached_nutrition_json`. Standard nutrition table layout, 14px values.

Explicitly OMITTED from public page: cost per portion, food cost %, selling price, chef notes, recipe tests, AI conversation, version history, Save/Recall controls, edit button.

## Publish Flow — Interaction Contracts

### Initial Publish (version picker dialog)

Trigger: "Publish recipe" button in builder header or card menu.

Dialog structure (`Dialog` component):
- Title: "Publish Recipe" (20px semibold)
- Body: "Choose the version to make public. This version will be visible to all users."
- `Select` component listing committed versions: label format "v{N} — saved {date}". Default selection: latest committed version.
- Sub-recipe prerequisite error: if any sub-recipe is unpublished, the dialog body shows an inline `Alert` (destructive variant): "The following sub-recipes must be published first: {name list}." Publish button is disabled.
- Footer: "Don't publish" (`variant="ghost"`) + "Publish recipe" (`variant="default"`)

### Unpublish (confirmation dialog)

Trigger: "Unpublish recipe" button in builder header or card menu.

Dialog structure:
- Title: "Unpublish Recipe" (20px semibold)
- Body: "This recipe will be removed from the public library. Your recipe data is unchanged."
- Footer: "Keep published" (`variant="ghost"`) + "Unpublish recipe" (`variant="destructive"`)

### Delete-while-published (blocked)

The delete action in the card menu is rendered disabled. On hover, a `Tooltip` reads: "Unpublish this recipe before deleting it." If the user somehow triggers delete via another path, show a `Dialog` with title "Cannot Delete Published Recipe" and body "Unpublish this recipe before deleting it." Single close button.

### Update to current version (one-click, no dialog)

Trigger: "Update to current" button in the builder header cue banner.

Behavior: sends `POST recipes/{recipe}/publish` with `version_id = current_version_id`. No confirmation dialog — the asymmetry is intentional (CONTEXT.md). On success, the cue banner disappears and the "Published" badge reflects the updated version. Show a `Sonner` toast: "Public version updated."

## Interaction Details

### Search debounce
- Delay: 300ms — matches established pattern in `recipes/index.tsx` (confirmed via codebase inspection).
- Partial reload: `router.reload({ only: ['recipes'] })` — same pattern; adapt prop name to `['library_recipes']` or match whatever the controller exposes.

### Pagination size
- 24 items per page — matches `RecipeController::index` paginate(24). Do not change.

### Filter set for public library
- Included: name/search, tag, cuisine, allergen, difficulty, max_total_time.
- Excluded: ingredient filter (not required by PUB-04 and not a useful browse signal for the public library).

### Nav label decision (Claude's Discretion)
- Use "Library" (short, clear, distinct from "My Recipes") with a `Globe` icon from lucide-react.
- Route prefix: `/library` (cleaner than `/recipes/public`, avoids confusion with the auth-gated `/recipes` prefix).

### Card link behavior
- `LibraryRecipeCard` links to `library/{slug}` via Wayfinder-generated `libraryShow({slug}).url`.
- Tab / keyboard navigation: the entire card is a `Link` wrapper (same pattern as `RecipeCard`).
- Hover state: `hover:shadow-md` transition on the inner `Card` (same as `RecipeCard`).

### Guest layout for public pages
- If `NavUser` crashes on null `auth.user`: create a `GuestLayout` component with a simple top bar containing the app logo, a "Library" link, and a "Sign in" link (right-aligned). Use `AppSidebarLayout` for authenticated sessions, `GuestLayout` for guest sessions.
- Detecting guest: `usePage().props.auth.user === null`.

## Accessibility

- Publish/unpublish buttons: explicit `aria-label` when icon-only. If text is present, no additional aria-label needed.
- Disabled delete menu item: `aria-disabled="true"` + `cursor-not-allowed`. Do NOT use the native `disabled` attribute on `<a>` or non-button elements.
- Loading skeleton grid: `aria-busy="true"` on the grid wrapper (same pattern as `recipes/index.tsx`).
- Dialog: use shadcn `Dialog` — Radix handles `aria-modal`, focus trap, and escape key natively.
- Published badge: not interactive, no role required.
- Public recipe page: `<h1>` for recipe name, `<h2>` for section headings, `<ol>` for ordered steps.
