## Spacing Scale

Declared values (multiples of 4 only — 8-point scale):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, inline badge padding |
| sm | 8px | Compact row internal padding, completeness dot gap |
| md | 16px | Card inner padding, dialog content padding |
| lg | 24px | Section gaps (queue header to table, review page section gaps) |
| xl | 32px | Page-level lateral padding on desktop |
| 2xl | 48px | Major section separators (nutrition panel / allergen panel dividers on review page) |
| 3xl | 64px | Page-level top spacing |

Exceptions:
- Nav badge pill: 6px horizontal / 2px vertical (matches existing shadcn Badge `px-1.5 py-0.5` default — not a custom override)
- Reject-note Textarea minimum height: 80px (to accommodate multi-line notes visually)
- Touch targets (submit/withdraw buttons): minimum 44px height on mobile

## Color

All values are OKLCH CSS custom properties from `resources/css/app.css`. Hex equivalents are provided for readability.

| Role | CSS Variable | Light Approx | Dark Approx | Usage |
|------|-------------|-------------|-------------|-------|
| Dominant (60%) | `--background` | warm off-white | dark charcoal | Page backgrounds, dialog backgrounds |
| Secondary (30%) | `--card` / `--sidebar` | warm light grey | dark near-black | Queue table rows, review page data panels, sidebar |
| Accent (10%) | `--primary` | near-black warm | warm off-white | Active submit/approve/withdraw CTAs only |
| Destructive | `--destructive` | red oklch(0.577...) | red oklch(0.396...) | Reject action button; delete button |

Accent reserved for:
- "Submit for Inclusion" primary button (owner CTA on ingredient detail page)
- "Approve" button on the per-submission review screen (moderator action)
- "Withdraw Submission" ghost/outline button uses `variant="outline"`, not accent fill

Muted/secondary tones (`--muted`, `--muted-foreground`) are used for:
- Completeness signal dots when a field is absent (greyed-out dot)
- "Contributed by" credit text below ingredient name
- Submission history timestamps and moderator names

Status badge color mapping (using shadcn Badge variants):
- `submission_status: private` — no badge shown (default state, no signal needed)
- `submission_status: submitted` — `variant="secondary"` (neutral; neither success nor failure)
- `submission_status: rejected` — `variant="destructive"` (red; matches `--destructive`)
- `submission_status: approved` (now official) — `variant="default"` (accent-filled; positive resolution)

## Interaction Contracts

### 1. Submit for Inclusion (Owner — Ingredient Detail Page)

**Location:** `ingredients/show.tsx` — alongside the existing `VerifyAction` component in the action area.

**States:**
- `submission_status: private` or `submission_status: rejected` → show "Submit for Inclusion" button (`variant="default"`, accent fill)
- `submission_status: submitted` → show `SubmissionStatusBadge` (Submitted) + "Withdraw Submission" button (`variant="outline"`) + frozen Alert banner
- `submission_status: approved` → show `SubmissionStatusBadge` (Official) + "Contributed by [name]" credit text; no action buttons

**Submit flow:**
1. User clicks "Submit for Inclusion"
2. System fires duplicate-check fetch against official ingredients by name
3. If ≥1 match: Dialog opens showing up to 5 matching official ingredients as a list; user must click "Submit Anyway" or "Cancel". If no matches: skip directly to step 4.
4. Confirmation Dialog: "Submit [Ingredient Name] for inclusion in the official library?" with "Submit for Inclusion" confirm button and "Cancel" dismiss.
5. On success: toast "Your ingredient has been submitted for review.", badge updates to Submitted, edit/delete buttons disabled by the frozen state (IngredientPolicy enforces this server-side; UI reflects via `can` flags).

**Withdraw flow:**
1. User clicks "Withdraw Submission"
2. Confirmation Dialog: "Withdraw your submission for [Ingredient Name]? It will return to private and you can edit and resubmit later." with "Withdraw" confirm and "Keep Submitted" dismiss.
3. On success: toast "Submission withdrawn.", badge removed, edit/delete buttons re-enabled.

**Frozen-while-pending:** When `submission_status: submitted`, the Edit button is hidden (not just disabled) and an Alert reads: "This ingredient is under review. Data is locked until a moderator decides." — `variant="default"` (neutral warning, not destructive).

### 2. Review Queue (Moderator — Admin Ingredients Page)

**Location:** `resources/js/pages/admin/ingredients.tsx` (stub → real queue)

**Layout:** Full-width Table. No filters or search on the queue — FIFO ordering is the primary navigation mechanism.

**Queue row columns:**

| Column | Width | Content |
|--------|-------|---------|
| Ingredient Name | auto | Name (name_cache); resubmission indicator badge if `submission_number > 1` |
| Submitter | 160px | User display name |
| Category | 140px | Subcategory name |
| Completeness | 120px | Three icon dots: nutrition / allergens / conversions |
| Submitted | 120px | Relative date (e.g. "3 days ago") |
| Action | 80px | "Review" link-button → review page |

**Resubmission indicator:** A `Badge variant="secondary"` reading "Resubmit #N" appears next to the ingredient name when `submission_number > 1`. Tooltip on hover: "Previously rejected — see rejection history on the review page."

**Completeness dots:** Three small `Tooltip`-wrapped circles (16px each, `gap-1`). Filled/green (`bg-green-500`) = data present; unfilled/muted (`bg-muted`) = data absent. Tooltip text: "Nutrition data complete" / "Allergens declared" / "Unit conversions added".

**Pending-count nav badge:** The "Ingredient Review" sidebar nav item renders a `Badge` with `pendingIngredientReviewCount` as its text when the value is > 0. Badge uses `variant="destructive"` to draw attention. Rendered only when the user has `review-ingredients` permission. Source: `usePage().props.pendingIngredientReviewCount`.

**Empty state:**
- Heading: "No pending submissions"
- Body: "All ingredient submissions have been reviewed. Check back later."
- No illustration; muted-foreground text centered in the table area.

### 3. Per-Submission Review Screen (Moderator)

**Location:** `resources/js/pages/admin/ingredients/show.tsx`

**Layout:** Two-column on desktop (≥ 1024px), single-column on mobile.
- Left / full-width: Ingredient data (Name, Category, Source; then Tabs: Nutrition | Allergens | Conversions — same tab structure as `ingredients/show.tsx`)
- Right sidebar (desktop) / below on mobile: Moderation panel (submission metadata, resubmission history, approve/reject controls)

**Moderation panel contents:**
1. Submission metadata: "Submitted by [Name] on [Date]" — `text-sm text-muted-foreground`
2. Submission number: shown only if `submission_number > 1` — Alert with prior rejection notes from all previous submissions in chronological order
3. "Approve" button (`variant="default"`, accent fill, full-width)
4. "Reject" button (`variant="destructive"`, full-width)
5. Optional note field for approval (collapsed by default; "Add a note (optional)" link expands a Textarea)

**Prior rejection history (resubmission context):** Rendered as a list of `Alert` blocks above the approve/reject buttons when `submission_number > 1`. Each block shows: "Rejected on [date] by [moderator]: [note text]". Uses `variant="default"` (neutral grey tone, not red — the rejection is history, not a current warning).

**Approve flow:**
1. Moderator clicks "Approve"
2. Dialog: "Approve [Ingredient Name] for the official library? This action is permanent. Approval also marks the ingredient as Verified." Optional notes textarea pre-expanded. Confirm: "Approve & Promote". Dismiss: "Cancel".
3. On success: redirect to queue (`/admin/ingredients`); toast "Ingredient approved and promoted to the official library."

**Reject flow:**
1. Moderator clicks "Reject"
2. Dialog: "Reject [Ingredient Name]? Please provide feedback so the submitter knows what to fix." Notes Textarea is required (server enforced; client shows inline error "Rejection notes are required" if submitted empty). Confirm: "Reject with Notes". Dismiss: "Cancel".
3. On success: redirect to queue; toast "Ingredient rejected. The submitter has been notified."

### 4. In-App Notification (Submitter)

**Delivery mechanism:** Shared prop `ingredientNotifications` (array of up to 5 unread DB notifications) injected via `HandleInertiaRequests`. Rendered in the app shell (sidebar or top of page).

**Notification entry display:**
- Approved: "[Ingredient Name] was approved and is now in the official library." — positive, no icon change needed.
- Rejected: "[Ingredient Name] was rejected. [Moderator's note]. You can update and resubmit." — shows the rejection note inline.

**Dismiss:** Notifications are marked as read when the user navigates to the ingredient detail page for that ingredient (server marks `read_at` on load). No dedicated notification centre — Phase 7-scoped only.

**Visual treatment:** Render as a stacked list of `Alert` components at the top of the sidebar or as a popover triggered by a bell icon in the nav — the specific placement is at Claude's discretion; must not block primary navigation. Keep it minimal: no red dot on the icon if there are no unread notifications; a `Badge variant="destructive"` count on the bell icon when unread count > 0.

### 5. "Contributed By" Credit (Post-Promotion)

**Location:** `ingredients/show.tsx` — rendered below the ingredient name heading when `ingredient.submission_status === 'approved'` AND `ingredient.user_id === null` (official).

**Appearance:** `<p class="text-sm text-muted-foreground">Contributed by [Chef Name]</p>` — no link (public contributor profiles are deferred).

## Destructive Actions Summary

| Action | Actor | Confirmation | Notes |
|--------|-------|-------------|-------|
| Withdraw Submission | Owner (submitter) | Dialog — requires explicit "Withdraw" click | Reversible: user can resubmit |
| Reject Submission | Moderator | Dialog — requires non-empty notes before confirm | Not permanently destructive for ingredient data; ingredient reverts to private |
| Approve & Promote | Moderator | Dialog — warns "permanent" | Irreversible in-app; only admin data-fix out of band |

Note: The existing "Delete Ingredient" action (`variant="destructive"` button on detail page) is blocked server-side while `submission_status: submitted` — no UI change needed beyond the frozen Alert already specified above. The delete button remains hidden (controlled by the `can.delete` flag which IngredientPolicy now returns false for submitted ingredients).
