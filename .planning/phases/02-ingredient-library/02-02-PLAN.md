Create `app/Support/Ingredients/IngredientImporter.php` (create the `app/Support/Ingredients/` directory). This class concentrates the upsert + sync logic so the three commands stay thin. It is a plain PHP class (use `php artisan make:class Support/Ingredients/IngredientImporter` if the command supports nested paths, otherwise create the file directly with namespace `App\Support\Ingredients`).

Public API:
```php
public function __construct() {} // promote nothing; resolves Allergen + Unit maps lazily
/**
 * Upsert ingredient rows in chunks of 500. Each $row is an array with keys:
 * source, source_id, category_id, name_cache, data_hash, + nutrition columns.
 * Returns the count of rows processed.
 */
public function upsertIngredients(array $rows): int
/** Reset verified=false for rows whose data_hash changed from the stored value. */
public function resetVerifiedForChangedRows(array $rows): void
/** Add a translation if (ingredient_id, locale) does not already exist. */
public function syncTranslation(int $ingredientId, string $locale, string $name): void
/** Sync allergen pivot from arrays of slugs. $contains and $mayContain are slug lists. */
public function syncAllergens(int $ingredientId, array $contains, array $mayContain): void
/** Insert a conversion row idempotently keyed on (ingredient_id, from_unit_id, source_ref). */
public function syncConversion(int $ingredientId, float $fromAmount, int $fromUnitId, float $gramWeight, string $source, ?string $sourceRef, ?string $modifier): void
/** Compute the md5 data hash from a nutrition payload array. */
public function dataHash(array $nutritionValues): string
```

Implementation rules:
- `upsertIngredients`: `array_chunk($rows, 500)`, each chunk inside `DB::transaction()`, call `DB::table('ingredients')->upsert($chunk, ['source', 'source_id'], $updateColumns)` where `$updateColumns` lists name_cache, data_hash, all 29 nutrition columns, and `updated_at`. Do NOT include `verified` in the update column list — verified is handled separately so unchanged rows keep their badge (RESEARCH Pitfall 2).
- `resetVerifiedForChangedRows`: after the upsert, run a second query — for each row, `DB::table('ingredients')->where('source', $row['source'])->where('source_id', $row['source_id'])->where('verified', true)->where('data_hash', '!=', $row['data_hash'])` ... but since the upsert already wrote the new hash, instead compute and store hashes BEFORE upsert: query existing `(source, source_id) => data_hash` for the batch, compare to new hashes, and reset `verified=false, verified_by=null, verified_at=null` for the source_ids whose hash differs. Document this two-pass ordering in a PHPDoc block.
- `dataHash`: `md5(serialize($nutritionValues))`.
- `syncTranslation`: `IngredientTranslation::firstOrCreate(['ingredient_id' => $id, 'locale' => $locale], ['name' => $name])`.
- `syncAllergens`: build `Allergen::pluck('id', 'slug')` map, build pivot array (contains wins over may_contain when both present), `$ingredient->allergens()->syncWithoutDetaching($pivot)` — use sync semantics that do not wipe a user's manual flags but do apply imported ones; for imports `sync()` is acceptable since imports own official ingredients.
- `syncConversion`: `IngredientConversion::firstOrCreate(['ingredient_id' => $id, 'from_unit_id' => $fromUnitId, 'source_ref' => $sourceRef], [...])`.

**ImportOpenFoodFacts** — `php artisan make:command ImportOpenFoodFacts`. Signature: `ingredients:import-off {--source-file= : path to an OFF tab-separated CSV} {--download : fetch the full OFF CSV gz on demand} {--country=Greece}`.
`handle()`:
1. If `--download`, stream the OFF CSV gz to `storage/app/tmp/` with `Http::sink()`, then stream-parse the gz without full decompression (RESEARCH Pitfall 3). Otherwise use `--source-file`.
2. Stream the TAB-separated CSV row by row. Filter to rows whose `countries_en` contains the `--country` value (default "Greece").
3. For each product: try to MATCH an existing ingredient — first by exact normalised English name (`product_name_en` lowercased/trimmed against `name_cache` lowercased), else treat as a new ingredient.

5. All operations upsert/firstOrCreate on stable keys so re-running is idempotent.
6. `$this->info()` summary; return SUCCESS.
