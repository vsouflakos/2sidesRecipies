## Cross-Phase Integration

All five major seams verified WIRED by the integration checker:

1. **Permissions (P1) → Recipe/AI/Publish/Moderation (P3/P5/P6/P7)** — The `getAllPermissions()->pluck('name')` fix (commit `8399453`) is confirmed live in `HandleInertiaRequests`. No other code path reads permissions the broken (direct-only) way. The three enforced permissions (`manage-users`, `review-ingredients`, `verify-ingredients`) gate the correct route groups.
2. **Ingredient Library (P2) → Recipe Metrics/Drafts (P3)** — `MetricsAggregator` resolves `ingredient_id` from draft JSON via batch-loaded `Ingredient` models; `GramNormalizer` uses `IngredientConversion` records. Correctly keyed by numeric PK.

Published snapshots store ingredient names as strings, so published recipes survive later ingredient renames/deletes.
4. **Ingredient Promotion (P7) → Recipe References (P3)** — convert-in-place promotion (`user_id=null`, `verified=true`, same PK) preserves all `recipe_ingredient_lines` FKs and draft JSON references; promoted ingredient becomes visible to all users and searchable in the builder. `IngredientPromotionTest` asserts this; 77/77 ingredient tests green.
5. **Localization (P1) → All later-phase UI** — `lang/en/app.php` and `lang/el/app.php` have full structural parity; all phase UI strings covered including Phase 7 additions.

## End-to-End Flows

The full user journey traced complete: register → create private ingredient → build recipe with it → record a test → use AI agent → publish → submit ingredient for moderation → moderator promotes it.

**Latent edge case (out of v1.0 scope, non-blocking):** hard-deleting a promoted official ingredient would null the `recipe_ingredient_lines.ingredient_id` FK while draft JSON retains the stale id, degrading the affected metrics line to a graceful data-gap banner (no crash). Hard-delete of official ingredients is not a v1.0 feature, and the `Ingredient` model uses `SoftDeletes` (which does not trigger the FK cascade).

## Tech Debt

**Total: 2 items across 2 phases — both non-critical, no v1.0 impact.**

**Phase 1 — Foundation**
- `create-recipes` and `manage-own-ingredients` permissions are seeded and granted to the User role but never enforced (full-codebase grep finds zero checks). Recipe creation / private-ingredient management is gated only by `auth`+`verified` middleware — functionally equivalent for v1.0, but the two named permissions are effectively dead code. Recommend wiring them into the relevant FormRequests/policies or documenting them as reserved for future granular gating.

**Phase 7 — Ingredient Moderation**
- `recipe_ingredient_lines.ingredient_id` uses `nullOnDelete()`; a hypothetical hard-delete of a promoted ingredient would degrade dependent recipe metrics gracefully (data-gap banner, no crash). Out of v1.0 scope.
